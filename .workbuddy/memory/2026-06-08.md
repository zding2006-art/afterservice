### Bug 修复：登录后 unauthorized

**根因 1**：Flask session cookie 缺少显式配置
- 修复：新增 SESSION_COOKIE_SAMESITE=Lax、SESSION_COOKIE_HTTPONLY=True、SESSION_COOKIE_SECURE=False、permanent_session_lifetime=86400

**根因 2（主因）**：前端 API 地址硬编码 `localhost:5859`，与用户访问的 `127.0.0.1:5859` 不同源
- 浏览器 SameSite=Lax 策略下，fetch 到不同 host 不携带 session cookie → 后端收到请求无 cookie → 401
- 修复：API 地址改为动态构建，使用当前页面实际 host (`${proto}//${host}:${port}/api`)
- 验证：PowerShell 全链路 (Login→check-auth→upload list) 通过

### Bug 修复：知识库月份错误 + 重复条目

**根因**：
1. `saveToKB()` 从全局变量 `allResults` 读取数据，但 `viewKB()` 渲染 KB 条目时不更新 `allResults`，导致保存按钮用的是旧上传数据
2. `renderMultiMonthResult` 原地修改 `data.results`/`data.summary`（翻译污染），污染了 `allResults`
3. 知识库表没有唯一约束，允许重复保存

**修复**：
- 新增 `currentDisplayedData` 变量，`saveToKB`/`saveMultiToKB` 从它读取（不再依赖 `allResults`）
- `viewKB` 从知识库查看时标记 `_isKBView: true`，隐藏「保存到知识库」按钮
- `renderMultiMonthResult` 使用局部变量 `translatedSummary`/`translatedResults`，不复写原始对象
- `saveToKB` 增加重复检测：保存前查询 KB 中是否有同年月条目，有则弹确认框
- 三语重复提示文案已添加

### Bug 修复：2月保存到知识库显示为6月

**诊断**：
- 后端 `/api/knowledge/save` API 测试正确（year=2025, month=2 → 保存成功，KB 列表显示正确）
- 前端 `currentDisplayedData` 数据流理论上正确，但可能存在旧的 KB 条目污染（之前 `allResults` bug 时期保存的错误数据）
- 也可能是 KB 列表刷新时机问题导致用户看到旧条目

**修复**：
- `saveToKB` 增加保存前确认弹窗（显示即将保存的月份、记录数、费用），用户可直观核对
- `saveToKB` 增加 console.log 诊断日志（输出 currentDisplayedData.summary + allResults.summary 对比）
- 后端 `kb_save_entry` 增加 logger.info 日志（记录收到的 year/month/label/records/cost）
- `saveToKB` 和 `saveMultiToKB` 保存后立即调用 `loadKB()` 刷新 KB 列表
- `saveMultiToKB` 补上 `loadKB()` 调用（之前缺失）
- i18n 新增 `kb.saveConfirmTitle`、`kb.saveConfirmHint` 三语文案

**后续建议**：知识库中可能存在之前 bug 留下的错误条目（如 6月被重复保存），建议用户核对并删除不需要的条目

### Bug 修复：月度列表「查看」后点保存知识库提示"没有可保存的分析数据"

**根因**：`viewMonth()` 调用 `renderUploadResult()` 渲染页面（含「保存到知识库」按钮），但从未设置全局变量 `currentDisplayedData`。用户点保存时 `saveToKB()` 检查 `!currentDisplayedData` 为 true → 触发 `kb.saveNoData` 错误提示。

**修复**：`viewMonth()` 中增加 `currentDisplayedData = { summary: detail, results: detail.analysis_data }`，与 `uploadFile()` 保持一致的数据流。
