## V1.7 月份归类优先级修正

### 背景
用户发现：文件名为 `202606`，但数据反馈日期为 `2025-06`，系统归类为 `2026年6月` 而非 `2025年6月`。

### 根因
`run_all_analysis` 在检测到 ≤1 个有效月份时，未使用 `detect_data_months` 的结果，直接用文件名解析的 `(year, month)` 进行分析。

### 修复内容
- `run_all_analysis`（`analyzer.py`）：在 `len(detected_months) == 1` 时，用数据中实际检测到的月份覆盖文件名月份
- 优先级：**数据记录日期 > 文件名日期**，无有效日期时回退到文件名

### 测试结果
- 文件名 `202606test.xlsx`，数据日期 `2025-06-*`
- `load_and_prepare` 返回：2026年6月（文件名）
- `detect_data_months` 返回：[(2025, 6)]
- `run_all_analysis` 最终归类：✅ 2025年6月

## V1.8 文件名与数据日期不一致的 UI 确认弹窗

### 背景
用户测试发现：文件名为 202606 但数据记录日期为 202506 时，应先让用户确认才保存。

### 修改内容
**analyzer.py**：
- `run_all_analysis` 在单月路径下，检测文件名月份 vs 数据实际月份是否一致
- 若不一致，在 `summary` 中附加 `filename_mismatch=True`、`filename_year`、`filename_month`

**app.py**：
- 单月上传路径：若 `filename_mismatch=True`，**不立即保存**，返回给前端带 `filename_mismatch` 标记的 JSON

**index.html（前端）**：
- 新增 `showFilenameMismatchConfirm(data)` 弹窗函数
- 弹窗显示「文件名」和「记录时间」两个标签对比
- 用户选 No → 放弃保存（toast 提示）
- 用户选 Yes → 调用 `/api/overwrite` 按记录时间保存；若该月已存在则再弹覆盖确认
